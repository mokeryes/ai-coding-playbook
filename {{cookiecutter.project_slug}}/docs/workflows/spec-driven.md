# SPEC-Driven Agentic Coding 工作流

> CLAUDE.md § 7 + § 8 的展开版。本项目所有新模块、重大变更必须按本流程执行。
> 与 [`.claude/skills/spec-driven-agentic-coding/SKILL.md`](../../.claude/skills/spec-driven-agentic-coding/SKILL.md) 同源,本文件是项目内的"操作手册版",skill 文件是 AI 启动时的"短指令版"。

---

## 核心思想

```
先定规格,再写代码;
边写边验,提交留痕;
AI 执行,人来把关。
```

每一步都有明确的**进入条件**和**离开条件**,不允许跳步,不允许"等等再补"。

---

## 6 阶段流程

```
┌─────────────┐  "开始写代码"  ┌─────────────┐  代码完成   ┌─────────────┐
│ 1. 方案确认  │ ────────────→ │ 2. 代码生成  │ ─────────→ │ 3. AI 自验   │
│  (写 SPEC) │               │             │            │ (语法/import)│
└─────────────┘               └─────────────┘            └──────┬──────┘
                                                                │自验通过
                                                                ▼
┌─────────────┐  用户说"提交"  ┌─────────────┐  6 项汇报   ┌─────────────┐
│ 6. Git 提交  │ ←──────────── │ 5. 收尾汇报  │ ←─────────  │ 4. 联调验证  │
│             │ (自检+commit) │             │ (按 SPEC § 9)│            │
└─────────────┘               └─────────────┘            └─────────────┘
```

**阶段编号说明**:本规范在前期版本里把"AI 自验"叫"阶段 2.5"。为了让进度可视化对称,
本版升格成独立"阶段 3",原"阶段 3 联调"顺移为"阶段 4",收尾 / 提交也获得明确编号。

---

## 阶段 1:方案确认

### 进入条件
用户提出新模块 / 新特性 / 重大变更需求。

### 必做动作
AI 在 `app/modules/<X>/SPEC.md`(或 `docs/specs/<feature>.md`)落档以下 13 项内容,用户审核:

| # | 项目 | 说明 |
|---|---|---|
| 1 | 背景与目标 | 为什么做、解决什么问题 |
| 2 | 输入 | 订阅哪些事件 / 读哪些表 |
| 3 | 输出 | 发布哪些事件 / 写哪些表 |
| 4 | 文件结构 | 目录树 |
| 5 | 关键函数签名 | 主要 API 草稿 |
| 6 | 数据库 schema | 完整 DDL |
| 7 | 事件 schema | BaseEvent + 特定字段 |
| 8 | 错误处理与降级 | 异常分类 + 处置 |
| 9 | **验收标准** | 联调期用什么命令 / 看什么数据 / 什么算成功 |
| 10 | **成功条件** | 正常路径下的预期行为 |
| 11 | **失败条件** | 已知会失败的输入 / 边界 |
| 12 | **回退策略** | 失败时下游如何兜底 |
| 13 | **默认值 / 本期不做** | 配置默认值登记 + 范围外明确划出 |

### 离开条件
用户**明确说出**下列触发词之一:

```
开始写代码
实现
继续实现
go ahead
write code
开干
```

### 禁止行为
- 用户只说"分析一下"、"先对齐"、"先改文档" → **不写代码**
- 用户没回应 → 不擅自开工

---

## 阶段 2:代码生成

### 进入条件
SPEC.md 完成 + 用户明确批准。

### 必做动作
- **先读代码,再改代码**(grep / 浏览相关文件,理解上下文)
- 优先沿用现有项目结构
- 改动严格按 SPEC.md 范围,不擅自越界
- 不主动重构无关部分
- 不擅自引入新依赖
- 不擅自改 CLI / API / UI 行为
- 不删除用户未确认的改动
- **遇到与 SPEC 冲突或超出 SPEC 范围 → 停下,在 SPEC.md 加 ADR 链接,等用户决策**

### 拆分顺序建议
```
解析层 → 状态模型 → 核心逻辑 → UI/CLI → 持久化 → 测试
```

### 测试同步(本项目策略)
- 纯逻辑函数(消息分类、schema 序列化、hash 比对、字符串处理):**鼓励**写 pytest / XCTest 等
- 外部依赖密集(SDK / DB / 网络):不写脆弱单测,留给联调阶段

---

## 阶段 3:AI 自验(本项目特有)

### 进入条件
代码生成完毕,交付前。

### 必做动作
AI 自跑下列检查中适用的一条或多条,在交付时汇报结果:

{% if cookiecutter.primary_language == 'python' -%}
- **语法检查(必做)**:`python -m py_compile <改动的 .py 文件>`
- **import 检查**(改了模块入口或共享基础设施时必做):`python -c "import <module>"`
- **DB schema 自检**(改了 schema 时必做):跑 init_db / migration 脚本
- **测试**(若有):`python -m pytest tests/`
{%- elif cookiecutter.primary_language == 'swift' -%}
- **编译检查(必做)**:`swift build` 或 `xcodebuild -scheme <X> build`
- **单元测试**:`swift test` 或 `xcodebuild test -scheme <X>`
- **SwiftLint**(若启用):`swiftlint`
{%- elif cookiecutter.primary_language == 'typescript' -%}
- **类型检查(必做)**:`npx tsc --noEmit`
- **Lint**:`npx eslint .`
- **测试**:`npm test`
{%- elif cookiecutter.primary_language == 'go' -%}
- **编译检查(必做)**:`go build ./...`
- **Vet**:`go vet ./...`
- **测试**:`go test ./...`
{%- elif cookiecutter.primary_language == 'rust' -%}
- **编译检查(必做)**:`cargo check`
- **Clippy**:`cargo clippy`
- **测试**:`cargo test`
{%- else -%}
- ⏳ 项目自填语言相关命令
{%- endif %}

### 没有执行权限时
明确告知:

> 我无法自跑 `<具体命令>`,请你执行验证 import / py_compile,如果失败贴报错给我修。

### 离开条件
保底语法 + import 不报错。完整集成测试不强求,留给阶段 4。

---

## 阶段 4:联调验证

### 进入条件
AI 自验通过,用户准备本地 / 测试环境验证。

### 必做动作
- 用户启动服务 / 喂数据 / 观察日志
- 按 SPEC.md § 9 验收标准**逐项核对**
- 出问题先 debug,**不为"换个写法"重写**
- 修复后重新走阶段 3 自验,再回到联调

### 离开条件
SPEC.md 验收标准全部勾选 ✅,可以收尾。

---

## 紧急沟通规则

实现过程中遇到以下 5 类情况,**必须停下询问用户**:

| # | 情况 | 例子 |
|---|---|---|
| 1 | **业务逻辑歧义** | 对一个数据应该走哪条分支不确定 |
| 2 | **跨模块边界争议** | 某功能放在 A 模块还是 B 模块 |
| 3 | **性能 / 成本权衡** | 某方案明显更贵但更稳 |
| 4 | **安全敏感操作** | 密钥处理、权限提升、数据删除 |
| 5 | **与 CLAUDE.md 冲突的需求** | 用户要求绕过本流程时 |

普通实现细节(变量名、内部函数拆分、日志措辞)**无需询问**,直接做。

---

## 阶段 5:收尾汇报

### 进入条件
阶段 4 全部 ✅ 通过。

### 必做动作
按 [`task-report-format.md`](task-report-format.md) 6 项格式汇报:
完成什么 / 改了哪些文件 / 新增测试 / 自跑验证 / 未验证项 / 下一步命令。

### 离开条件
用户说"提交" / "commit" → 进阶段 6;否则停在此处等用户决策。

---

## 阶段 6:Git 提交

### 进入条件
用户明确说"提交"或"commit"。

### 必做动作
按 [`pre-commit-checklist.md`](pre-commit-checklist.md) SOP:
1. `git status --short` + `git diff` 主动汇报
2. 7 项自检清单
3. 中文 conventional commits message(见 [`commit-style.md`](../conventions/commit-style.md))
4. 等用户最终确认 → `git commit`
5. (可选)推 GitHub — 用户决定

### 离开条件
commit 创建成功 + (可选)push 完成 + `.session_context.md` 状态字段更新到下个 SPEC 或清空。

## 进度可视化

> 让用户每次 AI 回答后,一眼看清当前流程位置。

### 触发条件

仅在 SPEC-driven 流程内追加进度块。**判断依据**:`.session_context.md` 的
`## 当前阶段(SPEC-driven)` 段中 `active_spec` 字段非 null。

闲聊 / 答疑 / 不在某个 SPEC 流程内时,**不加**进度块。

### 进度块格式

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 [SPEC: <模块/特性名>]  阶段 <X> / 6  <阶段名>
   <横向 6 大阶段链>
   └─ 当前阶段:
        ├─ 🟢 <子步骤已完成>
        ├─ 🟡 <子步骤进行中> ← 现在
        └─ ⚪ <子步骤待办>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 状态符号(色块 emoji,markdown 渲染时即彩色)

| 符号 | 含义 |
|---|---|
| 🟢 | 已完成 |
| 🟡 | 进行中(当前) |
| ⚪ | 待办 |
| 🔴 | 紧急停下(5 类必停下问题之一) |
| 🔵 | 回滚中(走 rollback-sop) |
| 🟣 | 自验失败重试 |

### 6 大阶段 + 子步骤

| 阶段 | 子步骤 |
|---|---|
| 1. 方案确认 | 1.1 复述 / 1.2 澄清 / 1.3 写 SPEC / 1.4 总结 / 1.5 等触发词 |
| 2. 代码生成 | 2.1 读相关 / 2.2 实现 / 2.3 同步纯逻辑测试 / 2.4 内部冲突停 |
| 3. AI 自验 | 3.1 语法 / 3.2 import / 3.3 schema / 3.4 测试 / 3.5 报告 |
| 4. 联调验证 | 4.1 用户启动 / 4.2 喂数据 / 4.3 SPEC § 9 逐项核对 |
| 5. 收尾汇报 | 5.1 6 项汇报 / 5.2 .session_context 更新 |
| 6. Git 提交 | 6.1 status / 6.2 diff / 6.3 7 项自检 / 6.4 中文 commit / 6.5 push? |

### 横向阶段链示例

```
🟢方案 → 🟡代码 → ⚪自验 → ⚪联调 → ⚪收尾 → ⚪提交
```

`retry_count ≥ 3` 时**强制升级**为紧急沟通(🔴),停下问用户。

### 状态权威来源

`.session_context.md` 的 `## 当前阶段(SPEC-driven)` 段。AI 在阶段过渡时
**主动更新**该段;用户也可以**手改**作为修正(如 AI 漂移)。

字段:

```markdown
## 当前阶段(SPEC-driven)

- **active_spec**: `app/modules/<X>/SPEC.md` 或 null(不在 SPEC 流程则 null)
- **major**: 1-6
- **sub**: 1.1 / 2.3 / ...
- **flag**: normal | interrupt | rollback | retry
- **retry_count**: 整数(失败重试计数)
```

### 多 SPEC 并发

**本期不支持**。若用户在 `active_spec ≠ null` 期间提另一个新 SPEC,AI 必须先警告:

> "你当前还在 SPEC X,先收尾(走完阶段 5/6)再开新 SPEC。如确实要切换,请手改
> `.session_context.md` 的 active_spec 字段。"

## 反模式(禁止)

| ❌ | 为什么不行 |
|---|---|
| "先写一版试试,边写边定" | SPEC 缺失 → 实现期反复返工,远比前期花时间写 SPEC 慢 |
| "这个改动小,不写 SPEC 了" | 小改动也常牵动多模块。**不写 SPEC 也至少要在 commit 信息里写清"为什么"** |
| "我先实现完,再补 SPEC.md" | SPEC.md 的价值是**事前约定**,事后补 = 装点门面 |
| "测试可以放下次再补" | 阶段 2 必须考虑测试。即便不立即写,也要在 SPEC § 9 验收标准 里说清未来怎么测 |
| "AI 自验跳过,直接联调" | 自验便宜,联调贵。语法错误本可在自验拦下 |
